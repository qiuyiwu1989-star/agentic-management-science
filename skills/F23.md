# F23 智能体企业架构模型（AEA）

## SKILL定位

AEA的核心价值是「共同语言」——技术人员和管理层之间的沟通桥梁。

**核心判断：协议层（MCP+A2A）是未来十年最重要的战略投资，类似AI版HTTP+TCP。**

治理层不是独立模块，必须内嵌在每一层。没有治理的架构只是技术堆砌。

AEA六层架构：
- L1 基础模型层
- L2 数据层
- L3 工具层
- L4 技能注册层
- L5 编排层
- L6 治理层（横贯L1-L5）

## 执行分析引擎（S1-S4四步法）

### S1：六层架构评估

**目标**：评估当前架构在AEA六层的成熟度

**各层评估维度**：

| 层级 | 评估要素 | 1分（缺失） | 3分（基础） | 5分（完善） |
|------|---------|-----------|-----------|-----------|
| L1基础模型 | 模型选型、版本管理、切换能力 | 单一模型硬编码 | 多模型可配置 | 模型路由+自动切换 |
| L2数据层 | 数据管道、向量库、知识管理 | 散落文件 | 基础向量库 | 统一数据治理 |
| L3工具层 | 工具注册、调用管理、权限控制 | 硬编码工具 | 工具列表可配置 | 工具Hub+权限 |
| L4技能层 | Skill注册、发现、版本管理 | 无Skill概念 | 基础Skill定义 | Skill Hub+生命周期 |
| L5编排层 | 流程编排、多Agent协作、路由 | 单Agent | 基础编排 | 多Agent编排+路由 |
| L6治理层 | 安全、审计、监控、合规 | 无治理 | 基础日志 | 全链路治理 |

**输出模板**：
```
S1架构评估：
- L1成熟度：__/5 关键差距：____
- L2成熟度：__/5 关键差距：____
- L3成熟度：__/5 关键差距：____
- L4成熟度：__/5 关键差距：____
- L5成熟度：__/5 关键差距：____
- L6成熟度：__/5 关键差距：____
- 总体评分：__/5
- 最薄弱层级：____
```

### S2：协议层战略评估

**目标**：评估MCP和A2A协议的战略价值和实施路径

**MCP（Model Context Protocol）价值矩阵**：

| 维度 | 当前状态 | MCP价值 | 优先级 |
|------|---------|---------|--------|
| 工具集成 | 每个工具单独对接 | 标准化接口，即插即用 | 高 |
| 数据连接 | 定制化数据管道 | 统一数据访问协议 | 高 |
| 跨模型兼容 | 绑定单一模型 | 模型无关的上下文传递 | 中 |
| 生态扩展 | 自建一切 | 接入第三方MCP生态 | 中 |

**A2A（Agent-to-Agent）价值矩阵**：

| 维度 | 当前状态 | A2A价值 | 优先级 |
|------|---------|---------|--------|
| Agent协作 | 硬编码调用 | 标准化通信协议 | 高 |
| 能力发现 | 手动配置 | 自动发现与注册 | 中 |
| 任务委派 | 单Agent处理 | 多Agent协作分工 | 中 |
| 跨组织协作 | 不可能 | 企业间Agent互联 | 低（长期） |

**输出模板**：
```
S2协议评估：
- MCP就绪度：____
- A2A就绪度：____
- 协议层投资优先级：____
- 预期ROI：____
- 风险：____
```

### S3：治理内嵌评估

**目标**：评估治理在各层的内嵌程度

**治理内嵌检查表**：

| 治理维度 | L1 | L2 | L3 | L4 | L5 |
|---------|----|----|----|----|----|
| 安全控制 | 模型安全过滤 | 数据脱敏 | 工具权限 | Skill权限 | 编排审计 |
| 可观测性 | 模型调用日志 | 数据访问日志 | 工具调用日志 | Skill执行日志 | 流程追踪 |
| 合规性 | 模型使用合规 | 数据合规 | 工具合规 | Skill合规 | 业务合规 |
| 成本控制 | Token成本 | 存储成本 | API成本 | 计算成本 | 总成本 |

**输出模板**：
```
S3治理评估：
- 治理覆盖度：____%（__/20格）
- 最薄弱治理维度：____
- 关键治理缺口：____
- 优先补全项：____
```

### S4：架构演进路线图

**目标**：制定从现状到目标架构的演进路径

**演进阶段**：

| 阶段 | 时间 | 目标 | 关键里程碑 |
|------|------|------|-----------|
| 夯实基础 | 1-3月 | L1-L3稳定 | 模型可切换、数据管道稳定、工具Hub上线 |
| 能力沉淀 | 3-6月 | L4-L5完善 | Skill Hub上线、编排引擎可用 |
| 治理内嵌 | 6-9月 | L6横贯 | 全链路监控、安全审计上线 |
| 协议升级 | 9-12月 | MCP+A2A | 协议层标准化、生态接入 |

**输出模板**：
```
S4演进路线：
- 阶段一（1-3月）：____
- 阶段二（3-6月）：____
- 阶段三（6-9月）：____
- 阶段四（9-12月）：____
- 关键决策点：____
- 最大风险：____
```

## 治理神经检查

### 意图管理检查
- [ ] 用户是要"画架构图"还是要"解决实际问题"？
- [ ] 架构评估是否与业务目标对齐？
- [ ] 是否存在过度架构化的倾向？

### 边界与升级检查
- [ ] 架构问题是否涉及组织变革（需升级到F07-F12）？
- [ ] 是否涉及商业模式调整（需升级到F06）？
- [ ] 技术选型是否超出AEA范围？

### 智能体主权检查
- [ ] Agent在架构中的自主权边界是否清晰？
- [ ] 治理是否过度限制Agent能力？
- [ ] 多Agent协作的主权分配是否合理？

---

## 质量自检

- [ ] 六层评估是否每层都有具体评分和差距说明？
- [ ] MCP/A2A评估是否基于实际业务需求而非技术跟风？
- [ ] 治理是否内嵌到每一层而非独立存在？
- [ ] 演进路线图是否有明确的时间节点和里程碑？
- [ ] 是否考虑了遗留系统的迁移路径？
- [ ] 投资建议是否与企业规模匹配？

## 典型误区

| 误区 | 正确理解 |
|------|---------|
| "架构就是技术选型" | 架构是业务、技术、组织的综合设计 |
| "MCP/A2A是未来，现在不用管" | 协议层投资越早越好，迁移成本随时间指数增长 |
| "治理是独立模块" | 治理必须内嵌到每一层，独立治理必然被绕过 |
| "六层都要一步到位" | 分层演进，每层稳定后再进下一层 |
| "架构评估是技术团队的事" | 架构是CEO级别的决策，需要业务+技术共同参与 |
