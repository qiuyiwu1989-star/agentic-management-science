# SKILL 标准输入/输出规范

> 所有SKILL文件的统一输入输出格式定义。确保SKILL可被标准化调用、结果可被结构化存储。

---

## 设计原则

1. **输入可采集**：每个SKILL需要的输入信息必须明确、可逐项收集，不依赖隐性假设
2. **输出可存储**：诊断结果必须是结构化的，可以存入用户profile或诊断报告
3. **过程可追溯**：执行步骤和判断依据需要被记录，支持事后复盘
4. **结果可比较**：同一SKILL对不同用户/不同时间的执行结果可以横向对比

---

## 标准输入格式

每个SKILL必须定义以下四类输入：

### INPUT-1：分析对象（必填）

```
对象类型：（行业 / 企业 / 岗位 / 产品 / 系统 / 个人）
对象名称：（具体名称，越精确越好）
对象描述：（当前状态、核心能力、市场位置等背景信息）
```

### INPUT-2：分析视角（必填）

```
发起方角色：（企业决策者 / 新进入者 / 投资人 / 个人 / 咨询师）
发起方背景：（与分析对象的关系、已有认知、核心关切）
```

### INPUT-3：触发问题（必填）

```
是什么具体问题触发了这次分析？
（必须是具体问题，不接受「帮我分析一下」）
示例：
  - 我们的护城河是否还有效？
  - 这个岗位应该优先升级哪些任务？
  - 我们的Agent系统应该用什么架构？
```

### INPUT-4：约束条件（选填）

```
时间跨度：（3个月 / 6个月 / 1年 / 3年 / 5年）
行业/地域限制：（如有）
已有假设：（发起方的初步判断，供SKILL验证或挑战）
```

---

## 标准输出格式

每个SKILL的输出必须包含以下五个部分：

### OUTPUT-1：核心判断（1-3句话）

```
最重要的判断是什么？
（不是「分析发现」，而是带有立场的判断）
```

### OUTPUT-2：分析结果（结构化）

```
根据SKILL类型不同，使用对应的输出模板：
- 分析型SKILL → 维度评分/矩阵/象限图
- 评估型SKILL → 等级/分数/差距
- 设计型SKILL → 方案/路径/架构
- 诊断型SKILL → 问题/根因/优先级
```

### OUTPUT-3：关键发现（3-5条）

```
用户可能忽视的洞见：
1. ____（与直觉相反的发现）
2. ____（隐藏的风险或机会）
3. ____（被低估或高估的因素）
```

### OUTPUT-4：行动建议（优先级排序）

```
下一步应该做什么？
1. [最紧迫] ____（时间框架：____）
2. [高价值] ____（时间框架：____）
3. [基础建设] ____（时间框架：____）
```

### OUTPUT-5：追问方向

```
基于本次分析，值得深入探索的方向：
- ____
- ____
（引导用户继续对话，不把分析终止在单次输出）
```

## SKILL文件元数据头规范

每个SKILL文件必须包含以下YAML front matter：

```yaml
---
name: SKILL_唯一标识名
version: 1.0
framework: 智能体管理学 · 模块X · 框架XX
type: 分析型 | 评估型 | 设计型 | 诊断型
description: >
  触发条件描述（什么时候应该激活这个SKILL）
  触发词：关键词1、关键词2、...
upstream_frameworks: 上游依赖的框架
downstream_frameworks: 下游关联的框架
governance_nerves: （如有）调用的治理神经
---
```

## 诊断报告标准格式

当SKILL被调用执行后，生成的诊断报告应遵循以下格式：

```markdown
# [SKILL名称] 诊断报告

## 基本信息
- 分析对象：____
- 发起方：____
- 触发问题：____
- 执行时间：____
- SKILL版本：____

## 核心判断
（OUTPUT-1）

## 分析结果
（OUTPUT-2，含图表/矩阵/评分）

## 关键发现
（OUTPUT-3）

## 行动建议
（OUTPUT-4）

## 追问方向
（OUTPUT-5）

## 执行日志
- S1：____（输入/输出摘要）
- S2：____（输入/输出摘要）
- S3：____（输入/输出摘要）
- S4：____（输入/输出摘要）
```
